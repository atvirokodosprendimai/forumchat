# rag

Semantic (vector) search over a community's **public** content ‚Äî the async
sibling of the FTS5 `search_fts` index (migration 00038). Gated by `RAG_ENABLED`.

## Why it exists separately from FTS

FTS5 is kept live **synchronously** by SQL triggers (00038). Vectors can't be:
producing an embedding needs an HTTP call to Ollama, illegal inside a trigger.
So migration **00039** triggers only *enqueue* a `(kind, ref_id, op)` marker into
`embed_outbox`; the `Worker` drains it, embeds, and upserts the vector store.
Realtime catchup is therefore eventually-consistent, bounded by
`RAG_WORKER_INTERVAL` (seconds, default 10).

```
content write ‚Üí SQL trigger ‚Üí embed_outbox row
                                   ‚îÇ  (Worker.tick every interval)
                                   ‚ñº
                 LoadDoc (re-read + visibility gate) ‚Üí chunk ‚Üí Embed ‚Üí Store.Upsert
```

## Shape (CQRS, like the rest of the codebase)

- `rag.go` ‚Äî domain types + the two swappable interfaces: `Embedder`, `Store`.
- `embedder.go` ‚Äî `Ollama` embedder (`/api/embed`, batch). bge-m3, 1024-dim.
- `store.go` iface lives in `rag.go`; `chromem.go` ‚Äî chromem-go impl. **Single
  collection** partitioned by a `community_id` metadata field (NOT per-community
  collections) so `DeleteByRef` needs no community and the qdrant port is a
  payload filter. chromem is an index only ‚Äî we pass precomputed vectors and use
  `QueryEmbedding`; its embedding func is a stub.
- `chunk.go` ‚Äî bge-m3 sliding window: `BodyTokens` primary (2800) + `Overlap`
  (400) bled in each side, ~400 slack under 4k. Token count is *estimated*
  (`estTokens` ‚âà runes/4); swap for a real tokenizer later without touching
  callers. Short content ‚Üí one chunk.
- `repo.go` ‚Äî `embed_outbox` queue (Dequeue/Ack) + the **loaders**. The loaders
  ARE the authorization boundary: each query's WHERE encodes what is
  community-public. **AI messages: only `role='assistant' AND status='done' AND
  thread.visibility='shared'`** ‚Äî private AI threads must never enter the
  community-wide index. Also `EnqueueAll` / `EnqueueCommunity` for reindex.
- `service.go` ‚Äî `Search` (read), `process` (one job), `ReindexAll/Community`.
- `worker.go` ‚Äî the drain loop. On the first process error it stops the tick
  (unacked) and retries next interval ‚Äî deliberate backpressure for "Ollama is
  down". Poison-row risk is accepted (content is plain text; a reindex clears it).

## Adding a content kind

1. a loader in `loaders` (repo.go) ‚Äî community resolved + visibility gated in SQL;
2. three triggers (insert/update/delete) in a new migration enqueuing the kind;
3. a backfill row in `enqueueAll` / `enqueueCommunity` and the migration backfill;
4. a `Kind*` const in rag.go.

The vector store has NO concept of users ‚Äî keeping private content out is the
loaders' job. Only ever index community-public rows (same rule as the internal
`list_issues` MCP tool ‚Äî see `internal/agent/mcpx`).

## Wiring (main.go)

Built before the agent block so `mcpx.Manager.RAGSearch` can expose the
`rag_search` tool (mapped `rag.Hit ‚Üí agent.SearchHit`). Worker started with the
other workers under `digestCtx`. `*rag.Service` satisfies the `Reindexer`
interfaces in `superadmin` and `admin` (assignment guarded so a disabled RAG
leaves the field a real nil, not a typed-nil).

## Reindex (after a backend switch, or to rebuild)

- **CLI** `forumchat-cli reindex [slug|all]` ‚Äî enqueue only (DB), no store touch.
  For a clean rebuild: stop server, delete `RAG_DB_PATH`, reindex, start.
- **/superadmin** "Reindex everything" ‚Üí `ReindexAll` (drop index + enqueue all).
- **/c/{slug}/admin** "Reindex this community" ‚Üí `ReindexCommunity`.
  The in-process buttons drop-then-enqueue; the CLI can't safely drop the
  chromem store while the server holds it open in memory.
